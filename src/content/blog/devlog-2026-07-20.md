> This is the first devlog on this site — a week-in-review compiled from what actually happened in GitHub, Azure DevOps, and Databricks, not from memory. Future editions will be daily. This one covers July 13–20, and it turned out to be a week with a plot: I spent Sunday teaching Databricks Genie to operate Azure DevOps, Monday night shipping a Campfire upgrade, and Tuesday discovering the free workspace under all of it had been cancelled.

---

## TL;DR

- **Genie can now drive Azure DevOps.** The `mosaic-ado-agents` project grew a "Genie lane": a spec-generated toolset of Unity Catalog SQL functions wrapping the ADO REST API, so a Genie space can list PRs, read work items, and tail build logs from plain SQL. Eight PRs landed through the GitHub → ADO → pipeline trunk in one day.
- **Campfire shipped its improvement batch** — real "Now Shipping" from live ADO pull-request data, persistent Delta storage, a change-keyed digest cache, and author-owned edits — merged at 2:01 AM UTC on July 15. The deploy that followed it failed two minutes later.
- **The free Databricks workspace died mid-week.** `Organization 4201007868433203 has been cancelled` — apps stopped, the event poller went silent after its 00:30 heartbeat on July 14. The response: migration runbooks, checklists, and paste-ready Genie space instructions. The prototyping phase is over; everything is packaged to redeploy wherever it lands next.

## Sunday: the Genie lane

The `mosaic-ado-agents` repo is the MCP-native rebuild of [ADO Companion](/blog/building-ado-companion) — agents authored as MLflow `ResponsesAgent`s, using Microsoft's ADO MCP server for tools, shipped as an Asset Bundle through the GitHub → Azure DevOps mirror trunk. By last weekend the agent fleet already existed: a Delivery Digest agent, a Request Intake agent with an approval-gated write path, a PR Reviewer with anchored comments and a clamped vote, plus eval suites and an event-fabric poller feeding them all.

This week's work was different in kind: **making the same ADO capabilities available to Genie**, Databricks' natural-language analytics interface. The mechanism is almost embarrassingly boring, which is why I like it — a **spec-generated set of Unity Catalog SQL functions** that wrap the Azure DevOps REST API, so anything that can run SQL (a Genie space, a notebook, a dashboard) can call ADO.

It took two attempts. The first cut hit friction, and the "plan B" that actually stuck (AB#63, PR 58) was **SQL functions over `http_request` with a UC connection** — live-verified against the real org before merging. One follow-up fix was needed for `ado_build_log_tail` on the serverless notebook engine (PR 59), which matters more than it sounds: tailing a failed build's log *from SQL* is the primitive that lets an analytics surface answer "why is the deploy red?"

Eight PRs went through the trunk on July 13 alone, every one following the same ritual: push to a GitHub branch, GitHub Action mirrors it to Azure DevOps, validate + AI-review pipelines run on the ADO PR, a human merges, and the deploy pipeline runs `databricks bundle deploy`. The validate pipeline went eight for eight; deploys to `dev` all succeeded. The AI review pipeline got cancelled a few times by fast merges — which itself produced a documented "fast-merge lesson" in PR 61. When your reviewer is a pipeline, merging too quickly is a race condition.

The rest of Sunday's batch was deliberately unglamorous: a rollout runbook, **paste-ready Genie space instructions**, a Genie kickoff prompt with the dev workspace's real values, and a milestone record for the docs acceptance flow. More on why that documentation push was well-timed in a moment.

Meanwhile the **event poller** — the job that watches ADO for new PRs and routes them to the reviewer agent — ticked along every 15 minutes, green on every run. Watermarks, idempotency, retry lane (a hotfix for the retry-lane CTE went out as PR 54 after the event fabric hiccuped). It would keep that perfect record right up until it stopped existing.

## Monday night: Campfire grows up

[Campfire](https://github.com/jpaladini/campfire) is the team wins-and-losses check-in app — a React SPA + FastAPI Databricks App where a ~25-person team logs wins, losses, help-needed, and learnings, and Claude Sonnet writes the team digest. The repo is barely a week old; this week it got the batch of improvements that makes an app feel *finished* rather than demoed:

- **Real "Now Shipping."** The panel that shows what the team is actually delivering now reads live Azure DevOps pull-request activity — merged-this-week, in-review, and blocked-over-2-days buckets per repo — instead of sample rows.
- **Real persistence.** Entries and settings land in Delta tables through the SQL warehouse, with the local-JSON fallback demoted to dev-only.
- **A smarter digest.** Digests are now recomputed only when the team log actually changes — a change-keyed cache — never per page view, and a scheduled bundle job pins the weekly digest every Monday morning. The digest also follows your filters, date range, and a personal scope now.
- **Ownership.** Entries are editable by their author, and help requests can be resolved.

That batch merged as ADO PR 40 at **02:01 UTC on July 15**. The deploy pipeline picked it up at 02:03. And failed.

## Tuesday: the sandbox dies

The failure wasn't a bug. The deploy log is unambiguous:

```
Error: cannot update resources.apps.campfire: Cannot create Databricks App,
  please try again later. (400 BAD_REQUEST)
Error: cannot create resources.jobs.campfire_digest_job: Organization
  4201007868433203 has been cancelled or is not active yet. (403 PERMISSION_DENIED)
```

The **free-edition Databricks workspace that hosted the entire dev environment was cancelled**. Both apps — `ado-companion` and `campfire` — show as stopped as of 01:13 UTC on July 14. The event poller's last heartbeat was 00:30. The Campfire improvement batch merged into a workspace that could no longer receive it.

Here's the thing: this was always going to happen, and the architecture was built for it. Back in the [original ADO Companion write-up](/blog/building-ado-companion) the constraint table called it: *Free Edition is fine for prototyping, not production — everything had to be config-swappable to another workspace.* Every piece of this system is a Databricks Asset Bundle with per-workspace targets and per-workspace secrets. Nothing environment-specific lives in code. The workspace is dead; the system is fine. It's sitting in git, waiting for a `databricks bundle deploy` pointed somewhere alive.

Which reframes Sunday's documentation push. The rollout runbook, the paste-ready Genie space instructions, the kickoff prompt, the migration checklist that went into Campfire as the very next commit after the failed deploy (ADO PR 62, still open) — that wasn't busywork, it was **packing**. When the sandbox evaporated, the week's output pivoted from *features* to *portability* within hours, and the honest answer is that the pivot was already half-done.

An opinionated aside: losing a free workspace with zero data loss and zero code loss is the best possible evidence that the bundle-first discipline was worth it. If any state had lived only in that workspace — hand-created jobs, hand-edited app configs, un-exported tables that mattered — this would have been a very different week. The only casualties were ephemeral: dev-tier Delta tables and a perfect green run history.

## What's next

The path is obvious and already written down, which is the point of runbooks: stand the bundles up in a fresh workspace, wire the per-workspace secrets, re-point the GitHub → ADO trunk's deploy target, and paste the Genie space instructions into a Genie space that talks to a live warehouse. Campfire's PR 62 — the migration checklist — is the open item that gates it.

And this devlog itself is part of the same story: it was assembled by an agent from the GitHub events API, Azure DevOps builds and PRs, and the Databricks jobs and apps APIs. If the week's work is going to be governed by pipelines and evals, the week's *retelling* might as well be too.
